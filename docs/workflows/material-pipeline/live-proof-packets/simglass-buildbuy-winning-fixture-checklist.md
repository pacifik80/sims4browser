## Required winning record

The first winning `Build/Buy SimGlass` fixture must record, at minimum:

1. stable reopened root
2. restart-safe object identity
3. companion resources that survived reopen
4. the exact evidence that kept the fixture in transparent-object scope
5. the exact evidence that allowed `SimGlass` to stay alive
6. the stronger object-side branches that were checked
7. why each stronger branch lost
8. why generic transparent fallback is too weak
9. explicit statement that the fixture passed the current promotion gate

## Minimum field checklist

### Identity block

- package name
- object name
- object-side identity or swatch anchor, if available
- reopened root

### Reopen block

- whether reopen was stable
- whether the path is restart-safe
- companion bundle that survived reopen

### Winning-branch block

- winning branch: `SimGlass`
- exact winning signal or surviving packet
- exact reason the result remains narrower than generic transparent fallback

### Losing-branch block

- why object-glass lost
- why threshold/cutout lost
- why `AlphaBlended` lost
- whether any contradictory signals remained and why they were weaker

### Promotion block

- explicit note that the current promotion gate passed
- explicit note that the current disqualifiers did not trigger

## Safe reading

- this checklist is stricter than the general transparent-object reopen checklist
- it exists because a winning `Build/Buy SimGlass` fixture carries a heavier burden than an ordinary transparent-object reopen

## Exact claim this packet is making

Exact target claim:

- the current workspace already knows how the first valid `Build/Buy SimGlass` win must be documented

Why this matters:

- it prevents the first true win from being recorded too loosely
- it turns the first valid promotion into a restart-safe precedent for later `SimGlass` fixtures
